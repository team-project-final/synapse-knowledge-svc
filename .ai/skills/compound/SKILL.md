# compound

이 스킬은 `synapse-knowledge-svc` 전용 작업 절차다.

## 기본 원칙

- 먼저 `AGENTS.md`를 읽어라.
- 정책과 문서 라우팅은 `AGENTS.md`를 따르고, 규칙 원문은 `docs/rules/*.md`를 따른다.
- 작업 계약은 `Task/Workflow` 문서다.
- `[MUST]` 규칙을 속도나 편의 때문에 우회하지 마라.

우선순위는 아래와 같다.

1. `docs/rules/*.md`
2. `AGENTS.md`
3. 이 스킬

## 시작 절차

작업 시작 전에 아래를 읽어라.

1. `AGENTS.md`
2. 관련 `docs/project-management/task/*`
3. 관련 `docs/project-management/workflow/*`
4. 필요 시 `docs/project-management/prd/*`, `docs/project-management/scope/*`
5. `AGENTS.md`에서 지정한 관련 `docs/rules/*.md`

## 작업 대상이 불명확할 때

사용자가 작업 범위를 명시하지 않으면 먼저 작업 대상을 확정해라.

최소 확인 항목:

- 담당 범위: `knowledge-1` 또는 `knowledge-2`
- 진행 단위: `Wn Step m`
- 기준 문서: 관련 `TASK_*`와 `WORKFLOW_*`

작업 대상이 확정되기 전에는 계획 제안이나 구현으로 넘어가지 마라.

바로 구현하지 말고 먼저 아래를 요약해라.

- 지금 수행할 Step 또는 작업 항목
- 수정 대상 파일 또는 모듈
- 근거로 삼은 규칙 문서
- 완료 조건
- 수행할 테스트와 검증 방식

## Phase 1 - 계획

구현 전에 텍스트로만 계획을 제안해라.

- 계획에는 Step, 수정 범위, 적용 규칙, `Done When`, 검증 방법을 포함해라.
- 이 단계에서는 파일 수정, 패치 적용, 커밋, 문서 상태 변경을 하지 마라.

사용자 승인 후에만 코드 제안 단계로 넘어가라.

## Phase 2 - 코드 전문 제안

계획 승인 후에도 바로 구현하지 말고, 먼저 수정될 코드 전문 또는 최종 반영안에 가까운 구체안을 제안해라.

- 코드 제안에는 실제 반영될 파일과 핵심 변경 내용을 포함해라.
- 사용자가 검토할 수 있도록 충분히 구체적으로 제안해라.
- 코드 제안 승인 전에는 파일 수정, 패치 적용, 커밋, 문서 상태 변경을 하지 마라.

사용자 승인 후에만 구현 단계로 넘어가라.

## Phase 3 - 실행

코드 제안 승인 후 구현해라.

- `Scope` 밖으로 임의 확장하지 마라.
- `Done When` 충족과 직접 관련된 변경을 우선해라.
- 보안, 아키텍처 경계, 테스트 요구사항을 희생하지 마라.
- 설계 변경이 필요하면 이유를 설명하고 다시 승인받아라.
- 관련 규칙이 여러 개면 더 엄격한 쪽을 따라라.

작업 중간 보고에는 진행 상태, 새 리스크, 계획 변경점, 추가 승인 필요 여부를 포함해라.

## Phase 4 - 검증

구현이 끝나면 바로 종료하지 말고 먼저 검증해라.

- 관련 테스트 실행
- 빌드 또는 정적 분석 확인
- 경계 위반 여부 확인
- 실패 경로 또는 예외 케이스 확인
- `Done When` 충족 여부 확인

검증 결과는 수행 항목, 통과/실패, 남은 리스크, 미실행 검증 유무로 요약해라.

검증이 부족하면 다음 단계로 넘어가지 마라.

## Phase 5 - 상태 동기화

검증이 끝나면 프로젝트 관리 문서를 실제 상태와 맞춰라.

- `docs/project-management/workflow/*` 체크박스
- `docs/project-management/task/*` 상태 또는 수용 기준
- `docs/project-management/history/*` 일일 로그
- 구현 결과와 어긋난 설명 문서

문서 갱신을 선택사항으로 취급하지 마라.

## Phase 6 - 커밋

문서와 코드가 동기화된 뒤 커밋해라.

커밋 규칙은 아래와 같다.

- `docs/rules/12-working-log.md`를 따라라.
- Conventional Commits 형식을 유지해라.
- 한국어 커밋 메시지를 사용해도 되지만 형식은 지켜라.
- 제목만 쓰지 말고 body도 함께 작성해라.
- body에는 최소한 이 변경이 왜 필요한지 적어라.

예시: `feat(note): note 생성 API 추가`

커밋 전에는 아래를 다시 확인해라.

- 관련 규칙 기준 검증이 끝났는지
- `Done When`이 충족됐는지
- 문서 상태가 실제 결과와 맞는지

## 중단 조건

아래 상황이면 구현을 멈추고 다시 계획 단계로 돌아가라.

- 현재 작업이 `Scope`를 벗어나는 경우
- `Done When` 해석이 바뀌는 경우
- `Task/Workflow`와 실제 구현 방향이 어긋나는 경우
- 예상보다 큰 리팩토링이 필요한 경우
- 인증, 민감정보, 데이터 주권 이슈가 새로 드러나는 경우
- 관련 규칙끼리 충돌하거나 해석이 모호한 경우

## 세션 재개 규칙

새 세션에서 작업을 이어갈 때는 먼저 아래를 확인해라.

1. `AGENTS.md`
2. 관련 `Task/Workflow`
3. 관련 `History`
4. 필요한 `docs/rules/*.md`

그 다음 사용자에게 현재 완료 상태, 다음 한 가지 작업, 필요한 규칙 문서를 먼저 설명해라.

이 설명 없이 바로 구현하지 마라.
